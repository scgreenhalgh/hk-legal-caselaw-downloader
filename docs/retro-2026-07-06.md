## What went well

- **TDD held throughout.** Every module landed with a failing-tests-first
  commit, then a paired feat/fix commit. Zero "just this once" bypasses.
  When a legis scraper canary revealed a schema bug on the 5-row canary
  we caught it before the full 1h47m run.
- **Reverse-engineering the SPA reliably.** For every new endpoint
  (`getlegisfiles`, `getcapversiontoc`, `gethoptfiles`, `gettreaty`,
  `getjudgment?lang=tc`) the pattern was: grep `chunk-c.js` for the
  method call, extract the params object shape, probe one live URL
  through a VPN endpoint to confirm the response shape, then write
  tests + code. Never took more than 3 probes to nail the call signature.
- **Primitive reuse paid.** ProxyPool, StructuredEventLogger,
  atomic_write, CheckpointDB, and the runner pattern from BulkScraper
  meant each new backfill was ~150 lines of code instead of ~1,000.
- **Parallel work while background scrapes ran.** The legis-history
  backfill and HOPT scraper had to serialise on the checkpoint DB lock,
  but I built the HOPT module during the history run and ran it
  immediately after the lock released.
- **Multi-agent workflow used precisely once**
  (`session-wrap-audit`) — three parallel forks for state / memory / artifact
  audits. Saved ~15-25k tokens of tool output in my context.

## What surprised us

- **Legislation ballooned 10×.** Started thinking 3,155 docs; ended with
  30,943 historical versions on top of the current 6,310.
- **Version IDs are NOT stable across `getcapversiontoc` calls.** Every
  TOC dump generates fresh section IDs even for unchanged sections. Only
  `pub_date` correlates across versions. Killed my initial "dedup by
  section id" scheme.
- **Only 17/134 Cap 1 sections unchanged since 1997** (12.7 %). Dedup
  potential was much smaller than I hoped, so I shipped the naive
  "one file per version" storage.
- **HKLII SPA uses different abbrs than the wire.** `bacpg` and `bahkg`
  both map to `hktba` at the backend. Nearly cost 240 misclassified rows.
- **Sample-based extrapolation for `has_translation` was 2.6× off.**
  My 200-file random sample estimated ~590 bilingual TC gaps; actual was
  1,517. Bilingual cases cluster by court/year (post-2010 hkcfa is
  ~universally bilingual, older hkfc is mostly EN-only), not uniformly.
- **HOPT `path=nd`.** Ten treaties silently disappeared from
  enumeration because the year segment was `nd` not `YYYY`. Only surfaced
  in the "missed anything?" audit.

## Twice-missed and thrice-missed things

**The user prompted a systematic audit after I twice-declared "full
coverage" that turned out to have gaps.** Findings across three
misses:

1. **Legislation history.** After `scrape-legis` I said "coverage 100 %".
   Only after the user asked "did we grab historical" did I realise we
   had the versions LIST but only the LATEST version's text. Fix:
   30,943 additional fetches over ~2 h.
2. **Case translations.** After `--lang both` scrape completed I said
   "bilingual coverage complete". Only after the user asked did I find
   that `--lang both` picked EN-when-both-exist and never fetched the
   TC counterpart. Fix: 1,516 backfilled sidecars.
3. **10 `nd`-year hkts treaties.** After HOPT completed I reported
   "1,129 downloaded". Only during the "did we miss anything" audit did
   I notice hkts totalfiles=266 vs on-disk 256. Fix: relax `_PATH_RE`
   to accept `nd|\d{4}`.

**Lesson:** Before claiming completeness, compare enum-time totals
against on-disk counts. When totalfiles > actual downloaded, investigate
the delta rather than assuming the missing entries "must be sparse
data". A one-line log warning "skipped 10 malformed paths for hkts"
during the original scrape would have caught this in real time.

**Implemented follow-up:** the enumerators now yield entries directly
so a skip is silent. Adding an explicit "n entries filtered because path
didn't match" counter to `parse_hopt_files_response` (and the parallel
`parse_files_response` for legislation) would prevent recurrence.
Small task, not shipped this session.
