# VPN Pool Architecture

The scraper reaches HKLII through 20 outbound proxies, each terminated inside a Docker container running [`gluetun`](https://github.com/qdm12/gluetun) on top of Private Internet Access OpenVPN. This chapter documents how the pool is provisioned, why every service is pinned to a specific PIA server, how DNS resolution stays inside the tunnel, and what the pool actually delivers in throughput. Consumers of the pool — per-proxy cookies, warm-up, IP-leak preflight — are covered in [Cookies, Sessions, and Warm-up](./07-cookies-sessions-warmup.md); the rationale for choosing gluetun in the first place lives in the [Decisions Log](./12-decisions-log.md).

## Why gluetun + PIA

Four properties made this the stack of choice, in decreasing order of importance:

1. **Namespace isolation via Docker.** Each gluetun container is a separate network namespace. A crash in one tunnel cannot affect the others, cannot bleed traffic onto the host route, and cannot leak the host's real IP. The scraper only ever sees each proxy as `http://localhost:<port>`.

2. **Native OpenVPN with PIA credentials.** Gluetun ships a PIA-specific provider driver (`VPN_SERVICE_PROVIDER: private internet access`, `VPN_TYPE: openvpn`) that authenticates with the same `OPENVPN_USER` / `OPENVPN_PASSWORD` the desktop client uses — no external key generation, no manual config wrangling. `docker-compose.yml:28-31` sets the four env vars and gluetun handles the rest. This is a hard requirement PIA does not meet the same way for WireGuard: PIA's WireGuard control plane needs configs generated externally via `pia-foss/manual-connections`, so we stayed on OpenVPN.

3. **DNS-leak safety by construction.** Gluetun runs Unbound inside the container with DNS-over-TLS resolvers, listening on `127.0.0.1:53`, and puts `nameserver 127.0.0.1` in `/etc/resolv.conf` of the container. The scraper never controls DNS — every hostname either goes to the HTTP proxy (which resolves inside the tunnel) or to Unbound (which does DoT inside the tunnel). See [DNS leak safety](#dns-leak-safety) below.

4. **Killswitch until tunnel up.** gluetun installs firewall rules that block egress from the container until the OpenVPN tunnel is established. If the tunnel drops, the HTTP proxy stops answering — the scraper sees `httpx.ProxyError` / `httpx.ConnectError`, not a home-IP leak. This gives us fail-closed behaviour by default and lets the code lean on IP-echo preflight (`ProxyPool.preflight` at `src/hklii_downloader/proxy_pool.py:264-290`) as a belt-and-suspenders check rather than the primary defence.

## Current topology

As of 2026-07-04, 20 healthy containers listen on `localhost:8888..8907`. The compose file is auto-generated by `scripts/expand_vpn_pool.py`; the excerpt below shows the shared config and the first two services (`docker-compose.yml:13-56`):

```yaml
x-vpn-common: &vpn-common
  image: qmcgaw/gluetun:latest
  cap_add:
    - NET_ADMIN
  devices:
    - /dev/net/tun:/dev/net/tun
  healthcheck:
    test: ["CMD-SHELL", "wget -qO- http://127.0.0.1:9999 || exit 1"]
    interval: 30s
    timeout: 10s
    retries: 3
    start_period: 30s
  restart: unless-stopped

x-vpn-env: &vpn-env
  VPN_SERVICE_PROVIDER: private internet access
  VPN_TYPE: openvpn
  OPENVPN_USER: ${PIA_USER}
  OPENVPN_PASSWORD: ${PIA_PASS}
  HTTPPROXY: "on"
  HTTPPROXY_LISTENING_ADDRESS: ":8888"
  HTTPPROXY_STEALTH: "on"

services:

  vpn-1:
    <<: *vpn-common
    container_name: hklii-vpn-1
    environment:
      <<: *vpn-env
      SERVER_REGIONS: "Singapore"
      SERVER_NAMES: "Server-10878-2a"
    ports:
      - "8888:8888"

  vpn-2:
    <<: *vpn-common
    container_name: hklii-vpn-2
    environment:
      <<: *vpn-env
      SERVER_REGIONS: "Singapore"
      SERVER_NAMES: "Server-10879-2a"
    ports:
      - "8889:8888"
```

Notable pieces:

- `HTTPPROXY: "on"` + `HTTPPROXY_LISTENING_ADDRESS: ":8888"` exposes gluetun's built-in TinyProxy inside the container on port 8888.
- `HTTPPROXY_STEALTH: "on"` strips the `Via:` and `X-Forwarded-For` headers that TinyProxy would otherwise append — the origin never sees a proxy fingerprint.
- Each container's internal `:8888` is host-mapped to a unique external port `8888 + (i - 1)`. `vpn-1` is on `localhost:8888`, `vpn-20` is on `localhost:8907`.
- The healthcheck hits gluetun's own control-plane HTTP endpoint on `127.0.0.1:9999`, which only returns 200 once the tunnel is up. `docker ps --filter health=healthy` is a reliable ready signal.
- `NET_ADMIN` + `/dev/net/tun` are the two capabilities OpenVPN needs to create the tun interface.

## SERVER_NAMES pinning

Every service sets both `SERVER_REGIONS` **and** `SERVER_NAMES`. The reason is subtle but load-bearing: gluetun's default behaviour when only `SERVER_REGIONS` is set is to **round-robin the pool of servers in that region on each (re)connect**. If the tunnel drops or the container restarts, gluetun picks a different physical server from the region's list. In practice that meant:

- Two containers in the same region could end up on the same physical server at once (wasted budget and no IP diversity gain).
- The exit IP for `vpn-3` after a restart is not the exit IP for `vpn-3` before the restart, so any log correlation or per-container speed tracking becomes meaningless.
- On short restarts, HKLII would see the same IP hop between multiple client fingerprints, which is exactly the pattern the [anti-detection strategy](./04-anti-detection-strategy.md) is trying to avoid (signal 10: deterministic per-proxy fingerprint stability breaks if the IP behind the fingerprint moves).

Pinning `SERVER_NAMES` to a specific physical hostname (e.g. `Server-10878-2a`, `hongkong402`) forces gluetun to always connect to that one server. The `(proxy_url, TLS profile, throttler state, HeaderRotator seed)` tuple documented in [Cookies, Sessions, and Warm-up](./07-cookies-sessions-warmup.md) then remains stable across restarts, which is what the deterministic per-proxy RNGs (`RequestThrottler(rng=random.Random(i))` at `src/hklii_downloader/proxy_pool.py:240` and `HeaderRotator(rng=random.Random(i + 1000))` at `:241`) assume.

Pinning is not "one server forever" — running `expand_vpn_pool.py` again refreshes the picks from a fresh `format-servers` dump. Server pins are stable within a run and easy to rotate between runs.

## expand_vpn_pool.py

`scripts/expand_vpn_pool.py` (264 lines) is the one and only supported way to generate `docker-compose.yml`. The legacy `scripts/generate_compose.py` reads `VPN_REGION_*` entries from `.env` and produces region-only services with no `SERVER_NAMES` pin; do not use it for anything new.

Pipeline:

1. **Query PIA catalogue.** `fetch_gluetun_servers()` at `scripts/expand_vpn_pool.py:44-53` shells out to `docker run --rm qmcgaw/gluetun format-servers -private-internet-access` and captures stdout. gluetun renders its own baked-in server list as a Markdown table — no external HTTP call, no third-party API — so the output is deterministic and offline once the image is pulled.

2. **Parse the table.** `parse_servers()` at `scripts/expand_vpn_pool.py:56-84` walks each line starting with `|`, splits on `|`, and produces `{region, hostname, server_name, tcp, udp, port_forward}` dicts. It detects TCP/UDP/port-forward flags by presence of `"✅"` in the relevant cell (`:80-82`). Header row (`region == "region"`) and the header separator (all cells consist of `-` or `:`) are skipped.

3. **Filter and pick.** `pick_servers_per_region()` at `scripts/expand_vpn_pool.py:87-104` walks the region list in order, keeps only servers matching each region, and slices to the first `per_region` entries. `--per-region all` skips the slice and takes every server; the current pool was generated with `--per-region all` (`memory/vpn-infrastructure.md:27`).

4. **Render compose.** `render_compose()` at `scripts/expand_vpn_pool.py:107-161` emits the YAML skeleton (shared anchor, PIA env block, `services:` header) then one indexed service per picked server, port-mapped starting at `BASE_PORT = 8888` (`:41`). Each service pins `SERVER_REGIONS` **and** `SERVER_NAMES` from the parsed row.

5. **Optional bring-up.** `--up` runs `docker compose up -d` and polls `docker ps --filter name=hklii-vpn- --filter health=healthy` every 15 s for up to 180 s (`scripts/expand_vpn_pool.py:222-252`). It exits `2` if fewer containers are healthy at the timeout than were provisioned. gluetun typically takes 60–90 s to go healthy — the 180 s ceiling has ~2× headroom.

6. **Optional test pass.** `--test` (implies `--up`) then runs `uv run python scripts/test_vpn.py` (`scripts/expand_vpn_pool.py:254-259`), which walks the four-phase test suite documented in `scripts/test_vpn.py:158-246`: home IP baseline, container health + DNS config, per-proxy exit IP + HKLII connectivity, PASS/FAIL summary.

Typical invocation for the current 20-pool:

```bash
uv run python scripts/expand_vpn_pool.py --per-region all --up
```

Default regions are the seven listed under [Regional distribution](#regional-distribution) — see `DEFAULT_REGIONS` at `scripts/expand_vpn_pool.py:31-39`. To pick a different region set:

```bash
uv run python scripts/expand_vpn_pool.py --regions Singapore,Macao --per-region 5
```

A region that PIA does not offer produces a `WARNING: no servers found for regions: [...]` line (`scripts/expand_vpn_pool.py:207-210`) but does not abort; the compose file is still written with whatever regions did match.

## PIA server catalog

The 2026-07-04 gluetun `format-servers -private-internet-access` dump exposed **476 physical servers across 165 unique region names**. The region list is snapshotted at `scripts/pia-regions.txt` (165 lines) as the canonical `SERVER_REGIONS` string reference — anything not in that file is not a valid region as far as gluetun and PIA are concerned. The APAC subset relevant to this project appears at `scripts/pia-regions.txt:50` (Hong Kong), `:60` (JP Tokyo), `:66-67` (Macao, Malaysia), `:93` (Singapore), `:97` (South Korea), `:100` (Taiwan), plus the multi-city and singleton candidates listed in [Expansion candidates](#expansion-candidates).

The gluetun catalogue is preferred over the raw `serverlist.piaservers.net` API because:

- gluetun dedupes physical servers by name and drops entries that its own OpenVPN driver cannot connect to, so what you pin is what you get.
- The Markdown table exposes TCP / UDP / port-forward capability per row, which the raw JSON does not surface cleanly.
- It works offline against a locally pulled `qmcgaw/gluetun:latest` — no third-party fetch inside `expand_vpn_pool.py`.

Refresh cadence: rerun `expand_vpn_pool.py` when either (a) PIA visibly changes their catalogue (new servers, decommissioned servers) or (b) you want to rotate off pinned servers between production runs. There is no automatic freshness check.

## DNS leak safety

Two independent paths need to be leak-safe: the scraper's DNS calls, and gluetun's DNS calls.

**Scraper side.** The scraper never resolves HKLII's hostname itself. It passes `http://localhost:<port>` to `httpx.AsyncClient(proxy=...)` (or to `curl_cffi.AsyncSession(proxy=...)` via `ImpersonateAsyncClient` at `src/hklii_downloader/impersonate_client.py:52-60`). Both httpx transports send the *hostname*, not a resolved IP, to the HTTP proxy in the `CONNECT` line (or the `GET` request-line for cleartext). Verified against httpcore source at `_async/socks_proxy.py:236` for the SOCKS path; the plain HTTP proxy path is even simpler — the entire URL is forwarded to the proxy. Result: no query for `www.hklii.hk` ever leaves the host's LAN.

**gluetun side.** Inside each container, `/etc/resolv.conf` reads `nameserver 127.0.0.1`. `scripts/test_vpn.py:84-99` (`container_dns()`) exec's `cat /etc/resolv.conf` in every container and checks for exactly that string. On the pool as configured, all 20 containers pass — verified as part of the 2026-07-04 canary. `127.0.0.1:53` in the container is answered by gluetun's embedded Unbound instance, which forwards each query to PIA's DNS-over-TLS resolvers *through the OpenVPN tunnel*. No plaintext DNS ever hits the host's physical NIC.

**Killswitch backstop.** If OpenVPN is not up, gluetun's firewall drops all egress including DNS, so any resolver call by anything inside the container simply times out. The `HTTPPROXY` on port 8888 also stops accepting connections (nothing behind it can talk to the internet). Scraper sees `httpx.ProxyError` and the ProxySession's circuit breaker fires (`src/hklii_downloader/proxy_pool.py:347-359`).

**Manual verification.** `scripts/test_vpn.py:243-246` prints the two commands to hand-check on the host:

```bash
sudo tcpdump -i en0 -n port 53 | grep -i hklii   # terminal 1
uv run python scripts/test_vpn.py                # terminal 2
```

No output from tcpdump = no leaks.

## PIA concurrent-device cap: removed 2023-2024

PIA historically enforced a 10 concurrent-device limit per account, which would have made a 20-container pool impossible on one subscription. That cap was lifted somewhere in 2023–2024 (memory/vpn-infrastructure.md:54). The current 20-container pool runs against a single PIA account with no issues — every container maintains its own OpenVPN session with the same `OPENVPN_USER` / `OPENVPN_PASSWORD` and PIA no longer counts them. The exact date is not documented on PIA's changelog, but the empirical evidence is: 20 concurrent healthy tunnels, all authenticated the same way, sustained through the 20-pool canary and prior 6-pool runs.

If PIA ever reinstates the cap, we would see gluetun healthchecks flip to unhealthy after ~10 tunnels came up (the 11th would fail auth), and the pool would need either a second PIA account or a switch to WireGuard, both of which are cheap workarounds.

## PIA auth token 24-hour expiry

PIA issues short-lived auth tokens from `www.privateinternetaccess.com` for OpenVPN authentication; these officially expire every 24 hours. Gluetun refreshes them automatically as part of tunnel maintenance — see the gluetun logs for lines like `INFO [openvpn] token expires in 23h58m, will refresh at ...`. There is no manual step and no observed correlation between token expiry timing and healthcheck flapping in the current setup.

For runs longer than 24 hours (a full-corpus scrape is currently estimated at 15–20 hours per [operations runbook](./11-operations-runbook.md), so this rarely bites), gluetun handles the refresh transparently. If token refresh ever fails — for instance if PIA rate-limits us or the OpenVPN cert rolls — the tunnel drops, the killswitch fires, and the scraper's circuit breaker takes the proxy out of rotation until cooldown.

## 20-pool canary

On 2026-07-04 the current 20-container pool ran a `--limit 100` canary. Full-court `hkfc` scoped scrape, all four content formats enabled (`html,txt,json,doc`), summaries + appeal history enabled, `--lang en` only. Result:

- **100 files in 52 seconds** — 2.3× faster than the 6-container pool's 2 minutes for the same 100 files.
- 100 / 100 downloaded, 0 failed.
- 6 / 6 randomly sampled proxies healthy at the end of the run (no circuit-breaker fires).
- Empty-content-with-doc fallback captured 55 `.doc` + 45 `.docx` files alongside 99 HTML.
- Appeal history 100 / 100 downloaded.
- Peak throughput ≈ 6,900 files/hour on the canary (100 files ÷ 52 s × 3600). Sustained throughput over a full run is lower once enumeration, per-item enrichment, and doc-fallback overhead average in — the production estimate is 15–20h wall clock for the full 4-court × 2-lang corpus.

The speedup is not linear in container count (20 / 6 = 3.3× vs measured 2.3×) because the request throttler (`src/hklii_downloader/proxy_pool.py:32-60`) adds burst gaps that partially serialize per-proxy work, and because the enumeration phase is single-tasked regardless of pool size (see [Scraper architecture](./09-scraper-architecture.md)). Adding more proxies helps download-phase parallelism but not enumeration wall-clock.

## Expansion candidates

The current pool leaves several PIA regions on the table. Adding them is a matter of running `expand_vpn_pool.py` with a larger `--regions` list and higher `BASE_PORT` allocations. Candidates:

**Multi-city (Australia).** PIA offers 5 separate Australian cities as region names: `AU Adelaide`, `AU Brisbane`, `AU Melbourne`, `AU Perth`, `AU Sydney` (`scripts/pia-regions.txt:6-10`). Together they contribute roughly 15 additional endpoints depending on the per-region count. Latency to HK is higher (~250–350 ms expected based on distance) but throughput would likely land in the same 60–100 KB/s band as the current pool's slower members.

**Asia singletons.** Each of these regions offers 1–5 named servers: `Bangladesh`, `Cambodia`, `China`, `India`, `Indonesia`, `Kazakhstan`, `Mongolia`, `Nepal`, `New Zealand`, `Philippines`, `Sri Lanka`, `Vietnam`. Aggregate potential is maybe 15–30 more containers if all are added.

**Not offered by PIA.** `Thailand` is conspicuously absent from `scripts/pia-regions.txt` — the 165-entry list has no Thai entry. If a Thai exit is required, the pool would need a second provider or a raw OpenVPN config from a Thai server operator; gluetun supports "custom" providers via `.ovpn` file mounts, but this is not currently wired up.

No expansion has been triggered yet because the 20-pool canary sustains enough throughput to fit the whole corpus inside a 15–20 hour window (extrapolated from the 100-file canary at 52 seconds peak). Doubling the pool would double health-monitoring complexity without an obvious win.

## Gap: no VPN health monitoring during long runs

The pool has *setup-time* verification (`scripts/test_vpn.py`, invoked by `expand_vpn_pool.py --test`) and *scraper-time* per-request circuit-breaker plus 50-request IP re-check (`src/hklii_downloader/proxy_pool.py:381-402`, documented in [Cookies, Sessions, and Warm-up](./07-cookies-sessions-warmup.md)). What it does not have:

- No independent monitor process watching `docker ps --filter name=hklii-vpn- --filter health=healthy` during a 15–20 hour production run.
- No alerting if a container flaps from `healthy` to `unhealthy` and back — gluetun will recover on its own, but the scraper only notices via failure counts, which are noisy.
- No SIGHUP-style refresh: if a container dies mid-run, the scraper's `ProxySession` (`src/hklii_downloader/proxy_pool.py:144-151`) will kill it after 5 consecutive failures and revive it 300 s later, but there's no automatic `docker compose restart` to bring the tunnel back if OpenVPN itself is genuinely broken.
- No coverage of the "all containers die at once" case — e.g. if PIA fires their auth token endpoint or a routing table gets clobbered. The scraper would eventually raise `AllProxiesDeadError` (`src/hklii_downloader/proxy_pool.py:20-22`) and abort, but with no orchestrated recovery.

This is flagged as follow-up in the [operations runbook](./11-operations-runbook.md) under known operational gaps. The minimum viable fix is a background loop that (a) polls `docker ps --filter health=unhealthy --filter name=hklii-vpn-` every ~5 minutes and (b) issues `docker compose restart <service>` for anything unhealthy for longer than one health-check cycle. That has not been implemented as of 2026-07-04.
